## 1. 设计目标

第一版的重点不是还原完整明末战争，而是以明末作为世界观入口，验证以下三个问题：

1. “塔防战斗 + 文官后勤”是否能形成有差异的策略体验。
2. “投入军资后获得更高战果回报”是否能形成清晰、健康的成长循环。
3. 上帝视角类 3D 表现能否在可控开发成本下提供足够的战场沉浸感。

长期剧情不限定于明末。奇观可以连接历史分歧、原创王国、异时空与未来遗迹，详细规则以[《开放式剧情框架》](./开放式剧情框架.md)为准。

设计原则：先完成单局闭环，再扩展长期养成；先做一条可玩的战线，再扩展多世界地图；先使用游戏内军资验证经济模型，不接入真实资金闭环。

## 2. 总体游戏结构

### 2.1 局外循环

```text
查看战线与剧情
  → 选择目标城池
  → 配置武将、兵种、文官和后勤
  → 进入战斗
  → 获得战果、资源和剧情进度
  → 强化军政能力
  → 解锁下一关卡
```

### 2.2 局内循环

```text
侦察敌军信息
  → 使用初始军资部署
  → 抵御波次并获得战中军资
  → 补充、升级或调整阵线
  → 应对关键敌人与首领波次
  → 保护城池并完成结算
```

### 2.3 胜负条件

- **胜利：** 消灭所有进攻波次，且核心城池耐久大于 0。
- **失败：** 核心城池耐久降至 0，或特定剧情保护目标被摧毁。
- **额外评价：** 剩余城池耐久、军资使用效率、关键目标存活和完成支线目标。

## 3. 战斗设计

兵种属性、骑乘与落马、武将重伤、弹药、装备耐久、阵型克制和战前整备的详细规则，以[《兵种与战斗规则》](./兵种与战斗规则.md)为准。本节保留总体战斗结构。

### 3.1 战场空间

- 使用 Godot 3D 场景搭建地形、城墙、道路、部署点和敌军入口。
- 镜头为倾斜俯视视角，支持平移、旋转和有限缩放。
- 关键交互对象使用描边、底盘或颜色提示，保证俯视状态下的辨识度。
- 首版采用固定路线与固定部署区域，降低自由寻路带来的平衡和性能成本。

### 3.2 部署规则

- 玩家选择单位后点击合法部署点进行放置。
- 不同部署点可限制地面、远程或特殊单位。
- 部署立即扣除军资；军资不足时不可部署。
- 普通士兵允许撤回并按一定比例返还军资；武将撤回可设置冷却时间。
- 单位自动索敌和攻击，玩家主要负责阵容、位置、时机与技能决策。

### 3.3 单位基础属性

| 属性 | 说明 |
| --- | --- |
| 生命 | 单位可承受的伤害总量 |
| 攻击 | 单次攻击的基础伤害 |
| 防御 | 减少受到的物理伤害 |
| 攻击间隔 | 两次攻击之间的时间 |
| 攻击范围 | 可锁定目标的范围 |
| 阻挡数 | 同时阻挡的敌人数 |
| 移动速度 | 可移动单位的行进速度 |
| 部署费用 | 部署所需军资 |
| 再部署时间 | 撤退或阵亡后再次部署的等待时间 |

### 3.4 首版兵种方案

| 单位 | 定位 | 特点 | 克制关系 |
| --- | --- | --- | --- |
| 刀盾兵 | 地面阻挡 | 高生命、高阻挡、输出较低 | 适合拦截普通步兵 |
| 弓手 | 远程输出 | 远距离持续攻击、生存较弱 | 适合处理轻甲与快速目标 |
| 长枪兵 | 地面输出 | 攻击距离略长、对冲锋单位增伤 | 适合反制骑兵或高冲击单位 |
| 守城武将 | 核心战力 | 属性较高，拥有鼓舞或范围攻击技能 | 用于关键路口和危急波次 |

### 3.5 首版敌人方案

| 敌人 | 定位 | 对玩家的考验 |
| --- | --- | --- |
| 流民军或轻装步兵 | 普通单位 | 检验基础阻挡与输出配合 |
| 轻骑或快速斥候 | 快速单位 | 检验多路线防漏和提前部署 |
| 重甲军 | 高耐久单位 | 检验集中火力与反甲配置 |
| 精锐头目 | 关键波次单位 | 检验武将技能和资源预留 |

具体敌军称谓应结合剧情立场和历史资料进一步校订。

### 3.6 波次结构

建议首个完整关卡包含 6～8 个波次：

1. 单路线基础教学。
2. 增加敌军数量。
3. 引入快速单位。
4. 开启第二入口或支路。
5. 引入重甲单位。
6. 混合兵种施压。
7. 短暂补给和整备。
8. 精锐头目或首领波次。

波次数据应由配置驱动，包括敌人类型、数量、生成间隔、入口、路线和开始条件。

## 4. 军资与后勤设计

### 4.1 战中军资来源

首版军资计算可采用以下结构：

```text
可用军资 = 关卡基础军资
         + 文官开局加成
         + 周期自然产出
         + 击败敌军奖励
         + 关卡事件奖励
         - 单位部署与升级消耗
```

### 4.2 战后收益

```text
战后奖励 = 关卡基础奖励
         × 难度系数
         × 完成评价系数
         + 首次通关奖励
         + 额外目标奖励
```

“返还比消耗更多”应表现为游戏内战果收益，而不是承诺现实现金收益。为避免玩家通过低风险重复关卡无限膨胀资源，重复关卡奖励可递减，或把关键成长资源放在首次通关和额外目标中。

### 4.3 文官方案

首版可以使用两名原型文官验证差异：

- **筹饷型：** 开局军资提高，适合快速建立防线。
- **转运型：** 每隔固定时间增加军资，适合持续时间较长的消耗战。

后续可扩展征粮、修城、征募、安民、情报和军械等方向。

## 5. 城池与关卡设计

朱由检身份、势力态势、首章时间线和完整剧情台词，以[《朱由检剧情需求调研》](./朱由检剧情需求调研.md)为准。

### 5.1 城池层级

| 类型 | 战略定位 | 玩法倾向 |
| --- | --- | --- |
| 京都 | 全局核心 | 最终防卫、政治事件、多线压力 |
| 王城 | 政权核心 | 首领战、关键剧情、极高难度 |
| 主城 | 区域中心 | 兵种与后勤解锁、区域控制 |
| 消耗战城池 | 前线节点 | 连续波次、补给受限、状态继承 |

### 5.2 第一版关卡结构

- **序章：山河将熄。** 崇祯十七年二月末至三月初，玩家从奇观中了解朱由检名义上统辖天下、实际军令和补给濒临断裂的处境。
- **第一关：昌平粮台。** 架空的京畿补给节点防守战，引入多兵种配合和快速敌人；胜利后粮秣与捷报进入北京，形成首个历史分歧点。
- **第二关：粮道危机。** 引入文官后勤和资源点保护。
- **第三关：京畿烽火。** 多入口、重甲敌人和精锐头目，保住通往北京的短暂军报通道，完成第一章闭环。
- **章末异象：世界之外。** 山河图继续展开，显现不属于大明与现实历史的世界切片，开放大明分歧线和原创世界线两个以上入口。

若开发资源有限，MVP 可先完成“主城守卫”一张综合地图，把其他内容缩减为教学阶段或同地图的不同关卡配置。

## 6. 表现与交互设计

### 6.1 视觉方向

- 推荐第一版采用低多边形或沙盘模型风格，减少写实建模和动画成本。
- 地形、阵营、兵种和可部署区域必须具有清晰的颜色与轮廓区分。
- 建筑和单位保持视觉 3D 效果，但不追求近距离第三人称级别精度。
- 历史氛围通过城防结构、旗帜、兵器、服饰轮廓和界面纹样体现。

### 6.2 战斗界面

- 顶部：当前波次、剩余敌人、城池耐久、暂停与倍速。
- 底部：兵种、武将卡片及部署费用。
- 侧边：军资数量、增长速度和文官效果。
- 战场提示：敌军入口、路线预警、可部署区域和伤害反馈。

### 6.3 操作方式

- 鼠标左键：选择、部署和确认。
- 鼠标右键：取消选择或打开单位操作。
- 鼠标滚轮：缩放镜头。
- 中键或键盘：平移、旋转镜头。
- 数字键：快速选择兵种或武将。

## 7. 技术架构设计

### 7.1 总体架构

```text
Godot 客户端
  ├─ 本地战斗模拟、画面表现、输入与 UI
  ├─ HTTP：登录、配置、存档、进度、战斗结算
  └─ WebSocket：实时通知、在线状态及后续多人能力预留
          ↓
Nginx
  ├─ HTTPS / WebSocket 反向代理
  ├─ 静态资源分发
  └─ 限流与基础安全策略
          ↓
Spring Boot 后端
  ├─ 账号与认证
  ├─ 玩家进度
  ├─ 游戏配置
  ├─ 阵容与养成
  └─ 战斗记录与结算校验
          ↓ MyBatis-Plus
MySQL（持久化数据） + Redis（缓存、会话、短期状态）
          ↓
Docker / Docker Compose（开发与首版部署）
```

### 7.2 客户端职责

- 场景、单位、动画、特效、音频和 UI 呈现。
- 单局战斗逻辑与输入响应。
- 加载服务端下发的版本化配置。
- 上传战斗摘要，接收服务端校验后的结算结果。
- 对网络中断提供重试和明确提示。

首版不建议逐帧同步战斗状态。塔防单局以本地模拟为主，服务端负责长期数据和结算校验，可显著降低开发复杂度与网络延迟影响。

### 7.3 服务端职责

- 账号注册、登录、令牌刷新和会话管理。
- 玩家档案、关卡进度、资源和阵容持久化。
- 单位、敌人、文官、关卡及波次配置发布。
- 战斗开始凭证、战斗摘要和奖励结算校验。
- 操作日志、异常记录、限流和基础防作弊。

后端分层建议：

```text
controller     接口接入与参数校验
service        业务接口
service/impl   业务接口实现
mapper         MyBatis-Plus 数据访问
domain/do      数据库对象
dto            请求与内部传输对象
vo             响应视图对象
config         框架与中间件配置
```

所有业务 Service 先定义接口，再在 `service.impl` 中提供实现。Controller 或其他 Service 仅注入接口类型。DTO、VO、DO、Record 等必须独立成文件，不嵌套在 Service 或 Controller 类中。

### 7.4 HTTP 与 WebSocket 边界

**HTTP 适用：**

- 登录与账号接口。
- 拉取游戏配置和公告。
- 查询及保存玩家进度。
- 创建战斗、提交战斗结果和领取奖励。

**WebSocket 适用：**

- 服务维护通知和配置更新提示。
- 在线心跳或跨设备登录提醒。
- 后续聊天、观战或多人协作能力。

首版若没有强实时业务，WebSocket 只搭建最小连接能力，不让其阻塞核心玩法交付。

### 7.5 MySQL 数据域草案

| 数据域 | 主要内容 |
| --- | --- |
| 用户 | 账号、凭证状态、创建时间、最后登录时间 |
| 玩家档案 | 昵称、等级、游戏内资源、当前章节 |
| 关卡进度 | 解锁状态、最高评价、首次通关时间 |
| 玩家单位 | 已解锁兵种、等级、养成状态 |
| 玩家角色 | 已拥有武将与文官、等级、技能状态 |
| 阵容 | 出战武将、兵种、文官和后勤配置 |
| 战斗记录 | 开始凭证、关卡、时长、结果摘要、奖励 |
| 配置版本 | 客户端版本、数值配置版本、发布时间 |

### 7.6 Redis 使用范围

- 登录会话、刷新令牌或令牌黑名单。
- 热点游戏配置缓存。
- 接口限流计数。
- 战斗开始凭证、短时幂等键。
- 在线状态和 WebSocket 会话映射。

不得把需要长期可靠保存的玩家进度仅存放在 Redis 中。

### 7.7 部署方案

首版使用 Docker Compose 管理以下服务：

- `nginx`：网关、HTTPS 和 WebSocket 代理。
- `backend`：Spring Boot 应用。
- `mysql`：业务持久化。
- `redis`：缓存与短期状态。

数据库和 Redis 使用独立数据卷；敏感配置通过环境变量或密钥文件注入，不提交到版本库。每次发布后仅保留本次镜像和最近一次可回滚镜像，确认服务健康后删除更旧的历史镜像。

## 8. 配置驱动设计

以下内容不应硬编码在战斗脚本中：

- 兵种和武将基础属性。
- 敌人属性和掉落军资。
- 关卡、入口、路线和波次。
- 部署费用、升级费用和冷却时间。
- 文官与后勤效果。
- 结算奖励、评价阈值和难度系数。

配置必须带版本号。客户端进入战斗时记录本局使用的配置版本，服务端结算时按对应版本进行合理性校验。

### 8.1 战斗功能模块边界

战斗系统按职责拆分为以下模块。模块之间只通过稳定的数据对象和事件交互，不直接读取其他模块的内部字段：

| 模块 | 核心职责 | 不应承担的职责 |
| --- | --- | --- |
| 单位模板模块 | 提供兵种基础属性、部署参数、可用形态和标签 | 不保存玩家武将成长，不执行战斗逻辑 |
| 武将模块 | 提供武将基础属性、适配度、熟练度、技能槽和装备槽 | 不复制兵种基础属性，不直接控制敌人 |
| 属性计算模块 | 汇总模板、武将、装备、状态和关卡修正，生成战斗快照 | 不播放表现，不修改持久化数据 |
| 技能模块 | 管理技能条件、目标、效果、持续时间、消耗和冷却 | 不在技能脚本中硬编码具体武将或兵种名称 |
| 装备模块 | 管理武器、防具、盾牌、马匹、弹药和耐久 | 不决定武将等级或兵种成长 |
| 状态模块 | 管理增益、减益、控制、形态切换和状态生命周期 | 不永久修改单位模板配置 |
| 战斗行为模块 | 负责移动、索敌、攻击、阻挡、受击和死亡/重伤 | 不直接读取数据库或玩家存档 |
| 战斗编排模块 | 负责波次、出生点、路线、胜负和战场事件 | 不包含具体兵种伤害公式 |
| 结算模块 | 生成战斗摘要、伤亡、奖励和耐久变化 | 不重新模拟客户端画面表现 |
| 表现模块 | 负责模型、动画、音效、特效、UI和提示 | 不作为伤害与状态判定的数据源 |

推荐依赖方向：

```text
配置仓库
  → 单位模板 / 武将 / 技能 / 装备定义
  → 属性计算器生成只读战斗快照
  → 战斗行为与状态系统消费快照
  → 战斗事件总线通知表现、统计与结算
```

表现层可以监听 `unit_deployed`、`attack_started`、`damage_applied`、`state_changed`、`unit_defeated` 等事件，但不得反向决定计算结果。

### 8.2 配置对象拆分

每类配置使用独立文件或独立数据表，禁止把完整战斗对象保存为不可拆分的大型 JSON：

| 配置对象 | 建议标识示例 | 主要内容 |
| --- | --- | --- |
| `UnitTemplate` | `unit.spear.infantry.t1` | 兵种基础属性、标签、部署参数、可用形态 |
| `UnitForm` | `form.spear.braced` | 骑乘/落马、远程/近战、列阵等形态属性覆盖 |
| `HeroTemplate` | `hero.defense_general` | 武将基础属性、技能槽、装备槽、初始适配 |
| `HeroProficiency` | `hero.defense_general:spear` | 武将对指定兵种的适配度、熟练度和经验 |
| `SkillDefinition` | `skill.inspire.v1` | 触发方式、目标筛选、效果列表、冷却和消耗 |
| `EquipmentDefinition` | `equipment.spear.standard` | 装备属性、耐久、标签和可装备条件 |
| `StatusDefinition` | `status.braced` | 状态标签、修正列表、叠加规则和结束条件 |
| `BalanceProfile` | `balance.mvp.20260723.1` | 全局公式常量、等级曲线、上下限和舍入规则 |

配置之间仅通过稳定 ID 引用，不通过中文显示名关联。显示名、描述和历史文本属于本地化资源，修改文案不得改变配置 ID。

### 8.3 属性计算管线

单位进入战斗或发生影响属性的事件时，由属性计算模块按照固定顺序生成快照：

```text
兵种/形态基础值
  → 固定值加成（装备、训练）
  → 同类百分比加算（熟练度、常驻被动）
  → 分类乘算（阵型、克制、技能）
  → 最终修正（关卡规则、难度、上下限）
  → 统一舍入并生成只读战斗属性快照
```

通用表达式：

```text
最终属性 = clamp(
    （基础值 + 固定值加成）
    ×（1 + 同类百分比加成之和）
    × 分类乘区一
    × 分类乘区二
    × 最终修正，
    属性下限，属性上限
)
```

约束：

- 同一来源、同一类型的百分比默认加算，不允许无限相乘。
- 克制、暴击、冲锋、技能增伤必须使用不同修正类型，并由 `BalanceProfile` 规定乘区。
- 血量上限变化时必须定义当前血量保持“固定值”还是“百分比”，首版统一保持当前血量百分比。
- 属性快照只在部署、形态切换、装备变化或影响属性的状态变化时重算；普通移动和纯表现事件不得触发全量重算。
- 伤害、治疗、状态命中和结算必须读取同一份快照，避免 UI、客户端模拟和服务端校验各用一套公式。

### 8.4 武将与兵种组合模型

武将不作为强化版兵种模板，而作为可挂接到部队的“指挥者组件”。组合后的战斗单位由以下内容构成：

```text
战斗单位 = 兵种模板
         + 当前形态
         + 武将指挥修正（可选）
         + 装备集合
         + 技能集合
         + 当前状态集合
         + 关卡修正
```

武将对兵种的影响分为三层：

1. **适配度：** 决定熟练度成长速度、等级上限和部分技能是否可用。
2. **熟练度：** 提供稳定的小幅属性增益，不替代兵种定位。
3. **技能与特性：** 提供有条件的战术变化，例如鼓舞、坚守、反冲锋或补给。

新增武将时只需新增武将、适配和技能配置；新增兵种时只需新增兵种、形态及必要装备配置。只有出现全新战斗机制时才新增通用效果执行器，例如“召唤”“位移”或“弹药转移”，不得为单个角色编写无法复用的专属计算分支。

### 8.5 技能效果组件化

技能由触发器、目标选择器、条件和效果列表组合：

```text
技能
  ├─ 触发：主动 / 被动 / 事件触发
  ├─ 目标：自身 / 所属编队 / 范围友军 / 范围敌军
  ├─ 条件：兵种标签、形态、血量、方向、冷却、资源
  └─ 效果：伤害、治疗、属性修正、添加状态、移除状态、资源变化
```

首版优先实现可复用效果：`Damage`、`Heal`、`ModifyAttribute`、`ApplyStatus`、`RemoveStatus`、`ChangeResource`。技能配置只声明参数，不直接持有场景节点引用。

### 8.6 扩展性验收

完成模块设计后，应使用以下改动验证扩展性：

- 新增一名武将，只新增配置和资源即可出战，不修改兵种类和属性计算器。
- 为现有兵种增加“列阵”形态，只新增形态、状态和效果配置，不复制整个兵种。
- 调整全体远程单位射程成长，只修改平衡配置或标签规则，不逐个修改技能。
- 新增一种装备耐久曲线，不影响没有该装备的单位。
- 服务端能够依据配置版本重建关键战斗参数，客户端配置过期时拒绝开始新战斗。
- 自动化校验能够发现重复 ID、无效引用、非法属性、循环形态转换和缺失本地化键。

## 10. 主要风险与应对

| 风险 | 影响 | 应对方案 |
| --- | --- | --- |
| 范围过大 | 第一版迟迟无法形成可玩版本 | MVP 只做一个区域、一张综合地图和最小阵容 |
| 3D 单位过多 | 帧率下降、动画和寻路成本高 | 固定路线、对象池、简化骨骼与材质、分级更新 |
| 军资正反馈失控 | 重复刷取造成经济系统膨胀 | 分离局内军资与长期资源，限制重复收益 |
| 历史表达争议 | 叙事失真或群体刻板化 | 明确历史幻想定位，关键内容进行资料核验 |
| 多世界内容失去统一感 | 各章节像互不相关的关卡合集 | 所有章节围绕“双向威胁、有限干预、胜利代价、世界回响”四项规则设计 |
| 联网架构过早复杂化 | 核心玩法开发受阻 | 战斗本地运行，服务端先负责存档和结算 |
| 客户端作弊 | 长期资源与进度被篡改 | 战斗凭证、配置版本、摘要校验、异常检测 |

## 11. 需要产品确认的设计决策

1. 第一章的具体地区、主城和主要历史人物。
2. 美术风格是否采用低多边形国风沙盘方案。
3. 玩家是否可以在部署后自由移动单位。
4. 武将技能采用主动释放还是全自动触发。
5. 局内军资与局外长期货币是否完全分离。
6. 首版是否要求离线可玩，联网仅用于同步存档。
7. 第二个开放世界优先选择永夜边城、赤潮群岛、铁路废土还是浮空王庭。
