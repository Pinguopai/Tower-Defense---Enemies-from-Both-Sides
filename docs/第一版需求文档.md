## 1. 项目概述

### 1.1 暂定名称

《两面来敌》

### 1.2 游戏定位

一款以上帝视角呈现、具有视觉 3D 效果的历史幻想题材塔防游戏。玩家通过部署武将与兵种、经营文官后勤体系、管理城池和战中资源，抵御多股敌对势力并逐步收复失地。

### 1.3 核心体验

- **观察历史奇观：** 玩家以现代人的身份旁观并逐步介入大明末期的历史进程。
- **花费资源改变战局：** 玩家消耗与“现实金钱”具有叙事映射的游戏资源，排兵布阵并改变战斗结果。
- **以战养战：** 成功防守后获得高于战前投入的回报，形成投入、作战、收益、强化、再战的正循环。
- **军政协同：** 武将和士兵负责战斗，文官和后勤负责资源生产与战场供给。
- **多方压力：** 玩家需要先后面对闯军、清军及其他地方敌对势力，在不同战线夺回城池。

### 1.4 目标平台

- 第一阶段：Windows PC。
- 后续可行性：根据 Godot 客户端适配情况评估 Linux、macOS 和移动端。

### 1.5 目标用户

- 喜欢塔防、策略布阵和资源经营的玩家。
- 对明末历史、架空历史或历史幻想题材感兴趣的玩家。
- 偏好短局战斗与长期养成相结合体验的玩家。

## 2. 世界观与剧情

### 2.1 背景设定

玩家是一名现代人，意外进入一处能够观测历史的“奇观”。奇观展现的时间线位于大明末期，李自成尚未进入北京之前。

起初玩家只能旁观。当玩家试图触摸奇观时，却发现自己能够通过消耗一种与现实金钱相对应的资源，在历史投影中部署士兵、任命将领并影响战局。

玩家成功抵挡第一波进攻后，发现奇观返还的资源竟然多于本次投入。出于好奇与对局势的责任感，玩家开始持续干预历史：整顿大明军政体系、反制闯军、抵御清军，并逐步收复原有土地。

### 2.2 第一版剧情范围

第一版聚焦“发现奇观—首次介入—守住主城—决定继续干预”的起始章节：

1. 玩家进入奇观并观察到一座即将遭受攻击的城池。
2. 玩家通过引导首次部署基础兵种。
3. 敌军发动第一波进攻，玩家完成防守。
4. 战斗结算时，玩家发现资源返还超过投入。
5. 玩家解锁文官后勤功能，通过经营获得下一场战斗的初始资源。
6. 玩家完成数场战斗，守住区域主城并获得通往下一地区的线索。

### 2.3 叙事原则

- 本作属于历史幻想和架空演绎，不宣称剧情是对真实历史的复原。
- 历史人物、地名、官职和事件在使用前需要进行资料核验。
- 避免把复杂历史冲突简单归结为单一民族或群体的善恶对立。
- “现实金钱”在第一版中优先作为剧情概念和游戏内资源映射，不直接等同于真实货币充值或现金回报。

## 3. 核心玩法需求

### 3.1 单局核心循环

```text
选择关卡
  → 配置武将、兵种与后勤
  → 获得初始军资
  → 观察敌军路线和波次
  → 消耗军资部署、升级或调动单位
  → 击败敌人并保护核心城池
  → 结算投入、战果与奖励
  → 强化军政体系并进入下一关
```

### 3.2 塔防战斗

- 战场采用上帝视角，可旋转、缩放和移动镜头。
- 美术表现为类 3D：使用 3D 场景和单位，或使用 2.5D 方案实现立体视觉效果；第一版原型优先采用 Godot 3D 场景。
- 敌军从地图上的一个或多个入口进攻。
- 玩家在指定部署区、道路节点或城防设施上布置单位。
- 敌军按路线向核心目标移动；部分敌人可攻击守军、城防或改变行进路线。
- 玩家通过合理配置近战、远程、阻挡和辅助单位完成防守。
- 每局由若干普通波次和一个关键波次构成。

### 3.3 军资系统

- 军资是单局内的主要消耗资源，用于部署兵种、任命武将、升级单位和发动技能。
- 初始军资由关卡基础值、文官能力和城池状态共同决定。
- 战斗中可通过时间产出、击败敌军、保护资源点和文官技能获得军资。
- 战斗结束后按照投入、剩余军资、守城结果和额外目标计算结算奖励。
- 第一版不接入真实货币消耗和现金返还，避免在规则、合规和经济模型未确认前形成错误实现。

### 3.4 武将与兵种

第一版至少提供以下战斗定位：

| 定位 | 作用 | 首版示例 |
| --- | --- | --- |
| 近战阻挡 | 阻挡敌人并承受伤害 | 刀盾兵 |
| 远程输出 | 在安全位置持续攻击 | 弓手 |
| 长柄反制 | 对骑兵或重甲单位具有优势 | 长枪兵 |
| 武将 | 高能力单位，拥有主动或被动技能 | 守城将领 |

武将是具有唯一身份、属性和技能的核心单位；普通士兵是可重复部署的基础战斗单位。

### 3.5 文官与后勤

- 文官不直接承担主要战斗职责，负责提高初始军资、战中军资产出、补给效率或战后收益。
- 玩家在战前选择出仕文官或后勤政策。
- 第一版至少实现两类效果：增加初始军资、增加周期军资产出。
- 文官体系与武将体系共同构成长期养成的“军政双线”。

### 3.6 城池体系

城池是关卡目标、长期经营节点和剧情推进载体。

- **京都：** 最高战略目标和核心剧情地点；失守会造成重大进度影响。
- **王城：** 君主或政权核心所在的城池，承载关键剧情和高难度决战。
- **主城：** 区域级核心城池，用于解锁兵种、文官、补给和相邻关卡。
- **消耗战城池：** 位于争夺区域或交通节点，以多波次、资源紧张和持续作战为主要特征。

第一版以一座区域主城及其周边战线为范围，不在 MVP 中完整开放全国地图。

## 4. 第一版功能范围

### 4.1 必须实现（MVP）

- 开场剧情和基础新手引导。
- 一个可旋转、缩放的上帝视角类 3D 战场。
- 一张完整可通关地图，建议另准备一张机制验证地图。
- 至少 3 个普通兵种和 1 名武将。
- 至少 3 类敌人，包括普通步兵、快速单位和高耐久单位。
- 敌人按波次和预设路线进攻。
- 单位部署、攻击、受击、死亡及撤离或回收。
- 初始军资、战中军资产出和部署消耗。
- 至少 2 名文官或 2 种后勤方案。
- 核心城池耐久、胜负判定和战斗结算。
- 玩家账号、基础存档、关卡进度和配置数据服务。
- HTTP 接口支持登录、存档、配置及结算；WebSocket 支持需要实时同步的状态或消息。
- Docker 化部署后端、MySQL、Redis 和 Nginx。

### 4.2 建议实现

- 战斗暂停、倍速和重新开始。
- 单位升级与简单技能。
- 图鉴或基础档案界面。
- 关卡三星或额外目标。
- 简单音效、背景音乐和战斗反馈。

### 4.3 暂不纳入第一版

- 大规模全国战略地图。
- 玩家对战、公会及大型多人在线功能。
- 自由建城或复杂城市模拟。
- 完整武将招募、装备、羁绊和抽卡系统。
- 真实货币投入、现金返还或具有投资属性的经济系统。
- 动态生成历史时间线。

## 5. 非功能需求

### 5.1 性能目标

- 目标平台在 1080p 下默认画质稳定达到 60 FPS；最低可接受为复杂波次下不低于 30 FPS。
- 第一版单场同时存在的战斗单位目标上限为 100 个，具体以原型压测结果调整。
- 普通 HTTP 接口在正常负载下的服务端 P95 响应时间目标小于 300 ms，不含客户端网络环境影响。

### 5.2 数据与安全

- 服务端负责账号、长期进度和奖励结算的最终校验。
- 客户端不保存数据库密码、服务端密钥等敏感信息。
- 登录凭证、接口鉴权和关键操作需要防重放与幂等控制。
- Redis 仅存放适合缓存或短期状态的数据，持久进度以 MySQL 为准。

### 5.3 可维护性

- 游戏数值、波次、单位和关卡配置尽量数据驱动。
- 后端业务 Service 必须先定义接口，再由 `impl` 子包中的实现类落地；Controller 和其他 Service 仅依赖接口。
- 业务实体、DTO、VO、DO、Record 等使用独立文件，禁止在业务类中嵌套定义多个类。
- 私有方法和可复用服务方法必须补充用途注释。

## 6. 第一版验收标准

满足以下条件可视为第一版核心玩法闭环成立：

1. 新玩家能够通过引导理解军资、部署、敌军路线和城池保护目标。
2. 玩家能使用至少三种兵种和一名武将完成一场多波次战斗。
3. 文官或后勤配置能明显改变开局或战中资源节奏。
4. 不同敌人需要玩家调整部署，而非单一单位即可无脑通关。
5. 战斗胜利后能够正确保存关卡进度并发放游戏内奖励。
6. 客户端退出并重新登录后能够恢复账号和关卡进度。
7. 服务端及其依赖可通过 Docker 环境完成部署和访问。

## 7. 待确认事项

- 游戏正式名称及对外宣传名称。
- “现实金钱”的最终表现：纯叙事概念、普通游戏货币，还是付费货币。第一版建议仅作为叙事映射。
- 是否严格使用真实历史人物与地名，还是采用架空化名称。
- 第一版主城及战役发生地点。
- 玩家直接操控单位的程度：仅部署、可调动，或可手动释放技能。
- 单局目标时长，建议首版以 10～20 分钟为验证区间。
- 美术方向：写实、国风写意、低多边形或棋盘沙盘风格。
- 第一版是否必须联网；当前架构按联网存档设计，但战斗可考虑本地运行、服务端校验结算。
